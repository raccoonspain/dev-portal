## D-001 · Ванильный JS без фреймворка на фронте

**Дата:** 2026-06-27 · **Статус:** принято

**Контекст.** Выбор фронтенд-стека для SPA-портала.
**Варианты.** React, Vue, Svelte, ванильный JS.
**Решение.** Ванильный JS + HTML + CSS, без сборщиков.
**Почему.** Портал — внутренний инструмент одного человека. Нет смысла в лишних зависимостях и build-шаге. Всё в одном файле `app.js` проще поддерживать и отлаживать.
**Последствия.** Нет hot-reload, нет типов, нет компонентной изоляции. При росте >2000 строк app.js — рассмотреть разбивку на модули.

## D-002 · SQLite только для чатов, проекты — файловая система

**Дата:** 2026-06-27 · **Статус:** принято

**Контекст.** Где хранить данные о проектах и шаблонах.
**Варианты.** SQLite для всего, файловая система, PostgreSQL.
**Решение.** SQLite только для чатов (история сообщений, session_id). Проекты и шаблоны — папки на диске.
**Почему.** Проекты и шаблоны — это реальные файлы, которые открываются в VS Code. Дублировать их метаданные в БД — избыточно. `annotation.md` прямо в папке проекта — документация там, где код.
**Последствия.** Нет поиска по метаданным проектов через SQL. Не критично для одного пользователя.

## D-003 · nohup вместо systemd (временно)

**Дата:** 2026-06-27 · **Статус:** принято, временное

**Контекст.** systemd-сервис `dev-portal` достиг лимита перезапусков и завис в статусе failed. Для сброса нужен `sudo systemctl reset-failed`, а sudo без пароля недоступен из Claude Code.
**Решение.** Запустить через `nohup node src/server.js` и сохранить PID в `portal.pid`.
**Почему.** Портал должен работать прямо сейчас. Простейший способ без sudo.
**Последствия.** При перезагрузке VPS сервис не поднимется автоматически. Нужно вернуться на systemd при первой возможности: `sudo systemctl reset-failed dev-portal && sudo systemctl start dev-portal`.

## D-004 · annotation.md как источник описания проекта

**Дата:** 2026-06-27 · **Статус:** принято

**Контекст.** Нужно хранить краткое описание проекта/шаблона для отображения в таблице портала. Старые шаблоны использовали `.description` (скрытый файл).
**Решение.** `annotation.md` в корне папки проекта/шаблона. Первая непустая строка — описание для таблицы.
**Почему.** Markdown-файл виден в VS Code, редактируется как обычный текст, не скрытый. Может содержать развёрнутое описание (остальные строки игнорируются порталом, но читаются человеком).
**Последствия.** Старый `.description` игнорируется. Для описания нужно создать `annotation.md`.

## D-005 · dev-portal как «закреплённый» элемент навигации с выделенным API-пространством

**Дата:** 2026-06-27 · **Статус:** принято

**Контекст.** Нужна быстрая точка доступа к самому порталу: скопировать путь, открыть в VS Code, просмотреть файлы, читать журнал (state/changelog/decisions). Рассматривалось два подхода.
**Варианты.**
- Добавить dev-portal как обычный проект в `/projects` и работать с ним через существующий интерфейс.
- Выделить отдельную вкладку в сайдбаре с жёстко закреплённым путём и выделенным API-пространством `/api/portal/…`.
**Решение.** Отдельная вкладка `🅰 dev-portal` выше `📁 Проекты`, выделенное пространство маршрутов `/api/portal/{open,tree,file,journal}`. Файловый браузер и markdown-рендер переиспользуются без изменений.
**Почему.** Портал — это не пользовательский проект, и смешивать его с `/projects` запутало бы список. Отдельный API-namespace даёт чистые маршруты без костылей вроде «зарезервированного» имени папки. Повторное использование существующей инфраструктуры (тип `'portal'` в `fbContext`) обошлось минимальными правками.
**Последствия.** Дерево файлов фильтрует `node_modules`, `.git`, `data` прямо на сервере. Журнал показывает только три вкладки (Состояние / История / Решения) — вкладка «Передача» не нужна для самого инструмента. При добавлении будущих «системных» объектов (например, `/templates`) этот же паттерн можно повторить.

## D-006 · Панель подробностей проекта по образцу страницы dev-portal

**Дата:** 2026-06-27 · **Статус:** принято

**Контекст.** В списке проектов нужен быстрый доступ к спецификации и журналу (CLAUDE.md, state, changelog, decisions) без перехода на отдельную страницу.
**Варианты.**
- Отдельная страница/роут для каждого проекта.
- Модальное окно поверх списка.
- Боковая панель-оверлей, как `file-browser` — открывается поверх контентной области.
**Решение.** Боковая панель (`#project-details`) с тем же позиционированием что и `file-browser` (position: absolute, left: 240px). Содержит заголовок, строку с путём и кнопками, вкладки CLAUDE.md / Состояние / История / Решения.
**Почему.** Паттерн уже существует в проекте (file-browser), переиспользование CSS и поведения минимизирует новый код. API `/api/projects/:name/journal` уже был — добавили только чтение `CLAUDE.md` из корня проекта. Интерфейс симметричен странице dev-portal, что снижает когнитивную нагрузку.
**Последствия.** Кнопка ❓ в каждой строке таблицы проектов. Если у проекта нет CLAUDE.md или docs/ — вкладки показывают «не найден» (штатное поведение). Тот же паттерн можно применить к шаблонам.

## D-007 · Деплой Битрикс24-приложений на тот же VPS через Caddy + PM2

**Дата:** 2026-06-27 · **Статус:** принято

**Контекст.** Нужен хостинг для Б24-приложений с HTTPS и постоянным доступом. Рассматривалось где и как размещать.

**Варианты.**
- Деплой на отдельный продакшн-VPS (дополнительный расход, сложнее CI/CD).
- Использовать сторонние PaaS (Heroku, Railway) — внешние зависимости, нет контроля.
- Деплоить на тот же VPS где dev-portal, используя уже стоящий Caddy.

**Решение.** Все Б24-приложения живут на том же VPS под доменом `b24.blackboxbegin.space`. Caddy маршрутизирует по путям (`/ap-simple-test*` → порт 3001, следующий → 3002, и т.д.). Node.js-серверы запускаются через PM2. Исходники — в `/projects/<проект>/`, статика — в `/var/www/b24/<проект>/` (Caddy работает от пользователя `caddy` и не имеет доступа к `/home/deploy/`).

**Почему.** Caddy уже настроен и выдаёт Let's Encrypt-сертификаты автоматически. PM2 обеспечивает автоперезапуск и сохранение процессов между ребутами. Минимальные дополнительные расходы — нет второго сервера. Путевая маршрутизация на одном домене масштабируется на любое количество проектов без новых SSL-сертификатов.

**Пожертвовали.** Изоляция: упавший процесс одного приложения теоретически влияет на доступность сервера для других. Нет staging/production разделения. Ресурсы сервера делятся между разработкой и «проодом».

**Последствия.**
- Порты 3000 (dev-portal) и 3001 (ap-simple-test) заняты; следующий проект берёт 3002.
- Для перезагрузки Caddy использовать `/usr/bin/caddy reload --config /etc/caddy/Caddyfile` (не `systemctl reload` — он может не применить конфиг).
- Б24 открывает iframe через POST — нужен Express с `app.post('/', ...)`, чистый `file_server` не работает.

## D-008 · Деплой Б24-приложений через PHP 8.3-FPM + OAuth server-side

**Дата:** 2026-06-28 · **Статус:** принято · **Отменяет (частично):** D-007 (Node.js + PM2 для Б24-приложений)

**Контекст.** D-007 предполагал Node.js + PM2 для Б24-приложений. На практике оказалось, что PHP — более естественный стек для Б24 local-app: готовые примеры от Bitrix, простой деплой без npm, не нужно держать постоянный процесс.

**Варианты.**
- Продолжать на Node.js + PM2 (Express, порты 3001, 3002, ...) — уже описано в D-007.
- PHP 8.3 + PHP-FPM через сокет — без постоянных процессов, Caddy передаёт запросы напрямую.
- Serverless / Docker — избыточно для текущего масштаба.

**Решение.** Б24-приложения пишутся на PHP 8.3, деплоятся в `/var/www/b24/<slug>/`, Caddy отдаёт их через `php_fastcgi unix//run/php/php8.3-fpm.sock`. OAuth (server-side code flow) реализуется средствами PHP без внешних библиотек.

**Почему.** PHP-FPM не занимает отдельный порт — порты остаются за dev-portal (3000) и code-server (8080). Меньше процессов = меньше слежки. Официальные примеры Bitrix24 на PHP, поэтому OAuth/REST-SDK хорошо документированы. `uri strip_prefix` в Caddy даёт чистые пути без slug-префикса.

**Пожертвовали.** Разнородность стека: портал на Node.js, приложения на PHP — нельзя переиспользовать код. `uri strip_prefix` убирает slug из `SCRIPT_NAME`, поэтому `init.php` определяет APP_URL без slug — нужно вручную поправить в `env.php` после первого запуска.

**Последствия.**
- PM2 для Б24-приложений больше не используется. Порты 3001+ свободны.
- Каждый новый Б24-проект: `bash deploy.sh` → правка Caddyfile → `/usr/bin/caddy reload` → ручная правка APP_URL в `env.php`.
- PHP-FPM работает от `www-data`, папка `data/` внутри проекта нужна с правами `0777`.

## D-009 · GitHub как внешний репозиторий + автоматический push после каждого коммита
**Дата:** 2026-06-28 · **Статус:** принято

**Контекст.** Все проекты (dev-portal, ap-simple-test, шаблоны) существовали только локально на VPS. При потере сервера код был бы утерян. Перенос на другой VPS требовал ручного копирования.

**Варианты.**
A) Оставить только локальный git — проще, но нет резервной копии и нет лёгкого переноса.
B) Подключить GitHub как remote — код в облаке, перенос через `git clone`.

**Решение.** Вариант B: созданы приватные репозитории на GitHub (`raccoonspain/dev-portal`, `raccoonspain/ap-simple-test`, `raccoonspain/_base_universal`, `raccoonspain/_b24-single-php`). SSH-ключ сгенерирован на VPS и добавлен в GitHub-аккаунт. В `scripts/snapshot.sh` добавлен автоматический `git push origin HEAD` после каждого коммита. Команды `/log`, `/decision`, `/handoff` и CLAUDE.md обновлены.

**Почему.** GitHub — страховка от потери кода при поломке сервера. Также упрощает перенос: достаточно `git clone` + перенести секретные файлы (`env.php`, `data/`). Автоматический push устраняет необходимость помнить про него вручную.

**Чем пожертвовали.** Каждый снимок теперь требует доступа к GitHub. Если нет интернета — push упадёт, но коммит сохранится локально, ошибки не будет.

**Последствия.**
- Все новые проекты при старте должны создавать GitHub-репозиторий и подключать remote.
- SSH-ключ `/home/deploy/.ssh/id_ed25519` привязан к аккаунту `raccoonspain` на GitHub.
- При переносе на новый VPS нужно сгенерировать новый SSH-ключ и добавить его в GitHub.

## D-010 · Контекстный чат с Claude по проекту + вставка изображений из буфера

**Дата:** 2026-06-28 · **Статус:** принято

**Контекст.** Разработчику нужен быстрый чат с Claude прямо в контексте проекта, не переключаясь на отдельный интерфейс. Дополнительно — возможность скидывать скриншоты для анализа.

**Варианты.**
- A) Открывать чат на отдельной вкладке браузера (уже был) — нет привязки к проекту, нет контекста.
- B) Встроить чат как сплит-панель справа от деталей проекта — проект и чат видны одновременно.
- C) Модальное окно поверх контента — теряется контекст файлов/журнала.

**Решение.** Вариант B: кнопка 🤖 в шапке панели деталей проекта и на странице dev-portal. При нажатии панель деталей сжимается до левой половины (CSS `right: calc(50% - 120px)`), справа появляется `#project-chat-panel`. Каждое открытие — всегда новый чат (не переоткрывает старый). Claude автоматически получает приветственный запрос с именем и путём проекта. Кнопка «+» в шапке чата — сброс и новый чат в рамках той же панели. Изображения из буфера: фронтенд захватывает `paste`-событие, показывает превью, отправляет base64 на сервер; сервер сохраняет в `/tmp/`, передаёт путь Claude через инструкцию «прочитай файл через Read», удаляет файл после ответа.

**Почему.** Сплит-панель — единственный вариант, где видны и документация проекта, и чат одновременно. Новый чат при каждом открытии — чистый контекст без загрязнения предыдущими разговорами. Передача изображения через файловую систему (а не через API) позволяет работать с OAuth-сессией Claude без API-ключа: Claude CLI умеет читать PNG/JPG через `Read` tool.

**Чем пожертвовали.** История чата не сохраняется между закрытием и открытием панели (намеренно). Старые чаты по проекту накапливаются в БД под темой = название проекта — доступны через основной список чатов. Изображения не сохраняются в истории: при перезагрузке страницы миниатюра не отображается (видно только текст сообщения).

**Последствия.**
- `POST /api/chat` принимает опциональное поле `image` (base64, до 10 МБ после увеличения лимита `express.json`).
- Временные файлы изображений — `/tmp/portal-img-*.{png,jpg,gif,webp}`, удаляются в `proc.on('close')`.
- `pcpStreamToElement` — общий SSE-хелпер для инициализации и пользовательских сообщений (DRY).
- При добавлении нового типа чат-панели нужно реализовать аналогичный paste-хендлер и передавать `image` в `pcpStreamToElement`.
